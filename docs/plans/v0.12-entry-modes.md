# v0.12 — composable modules + research rigor (Claude-Science-informed)

Motivation: rigorloom today has ONE golden path (topic -> full report). Real
usage enters at many points. Rather than hardcoding N modes, define a small
catalog of MODULES with typed input/output contracts; a deterministic resolver
composes the minimal module chain from (artifacts you HAVE) -> (artifact you
WANT). Named "modes" are just saved compositions (aliases), not separate code
paths. Research rigor adopts the two ideas Claude Science / OpenScience center
on that we lack: a per-claim evidence ledger and formalized actor-critic roles.

## Module catalog (typed contracts)

Each module: `consumes -> produces`, its gate(s), OS constraint. Modules map
~1:1 onto existing stages/scripts; this is a contract layer, not a rewrite.

| Module | Consumes | Produces | Gate | OS |
|---|---|---|---|---|
| topic_select | constraints | topic candidates | human pick | any |
| research | topic (+constraints) | evidence pack, claims.yaml, source cache entries | evidence completeness | any |
| claim_extract | content.md | claims.yaml (from prose) | — | any |
| retro_research | claims.yaml (unevidenced) | evidence pack + backfilled ledger | per-claim support check | any |
| design | topic + evidence | outline/design doc | human gate (supervised) | any |
| data_sim | design | data, figures, seeds | deterministic verdict (existing) | any |
| write | design + evidence + ledger | content.md | no unledgered claims | any |
| humanize | content.md | content.md (voice) | fact-invariance check | any |

## Resolver

`pipeline_ctl compose --have <artifacts> --want <target>` (and `mode:` in
request.yaml as sugar): deterministic — walk the contract DAG backwards from
the target, include a module iff its product is missing from `--have`,
topologically order, emit the stage plan into PIPELINE.md like today. No
search/planner heuristics; the DAG is small and acyclic. Intake validates the
contracts of the first scheduled module (e.g. composing from `have: evidence`
demands the evidence pack actually parses).

Named aliases (docs + request.yaml sugar) = the compositions users asked for:

- full-report: topic -> [research, design, data_sim, write, humanize,
  content_verify, assemble, render_proof, submit_verify]
- conditions-only: constraints -> topic_select + full-report
- pre-researched: evidence -> full-report minus research (supplied evidence
  still passes source verification + enters the ledger)
- backfill: content.md -> [claim_extract, retro_research, content_verify, ...]
- verify-only: doc -> [content_verify, submit_verify] (+ render_proof optional)
- assemble-only: content.md + form -> [assemble, render_proof, submit_verify]
- form-edit: report + request -> [edit, submit_verify]
- form-convert: report A + form B -> [content_extract, assemble(B),
  content-hash parity gate, submit_verify]
- taste-mine: corpus -> [form_extract, style_extract] -> draft packs for
  operator review

Adding a future mode = adding an alias (or nothing — resolver already
composes it), not new pipeline code.

## Research rigor

R1. **Claim ledger** (core): research/claim_extract emit `claims.yaml` — every
factual claim bound to evidence (source id, locator, quote/figure). write may
not introduce unledgered factual content. New deterministic `check_claims`
(runs inside content_verify): body numerals/citations must trace to ledger
entries; unledgered -> WARN, ledgered-but-source-missing -> HARD. Narrows the
known "real DOI on an unsupported claim" hole: the ledger records WHAT each
source supports, giving the LLM audit layer a machine-checkable anchor.
backfill (retro_research) is built on this ledger.

R2. **Write-through source cache**: research fetcher records DOI/ISBN
verification into `<PROFILE_ROOT>/cache/sources/` at fetch time (fetcher-
written only). check_sources' unverified-WARN becomes the exception.

R3. **Actor-critic formalization**: playbook-level writer/critic roles per
generative module (draft vs citation-fidelity critic in write; assembly vs
format critic in assemble). Critics produce findings; deterministic gates stay
the only verdict authority.

R4. **OpenScience evaluation** (cheap, optional): Windows-capable, Apache-2.0,
MCP-based. Evaluate as a research front-end feeding our evidence pack + ledger.
Adapter only; no dependency.

## OS parity contract

Capability = per-module, so any composition's OS ceiling is computable:
min over its modules. Publish the module OS column + per-alias derived matrix;
CI runs one synthetic smoke per alias on ubuntu+windows. Submission-grade
output remains Windows+Hancom (honest); everything else OS-neutral or tiered.

## Waves

- **W1 module contracts + resolver**: contract manifest (modules.yaml), compose
  command, intake validation, aliases full-report/pre-researched/verify-only/
  assemble-only (all ~formalization of existing paths), per-alias CI smokes,
  capability matrix doc.
- **W2 checker_base refactor** (prereq for new checkers): shared verdict/exit/
  extraction module, composition unification (v0.11-Z5 structural-debt list).
- **W3 claim ledger + write-through cache** (R1+R2): claims.yaml schema,
  check_claims, fetcher cache writes, backfill + conditions-only aliases.
- **W4 transform modules**: edit hardening, content_extract + form-convert
  (content-hash parity gate), form/style_extract -> draft packs (taste-mine).
- **W5 campaign**: 5-10 real reports across aliases (incl. verify-only over
  past submitted reports — cheap, high-signal). Collect false-positive/
  resume-failure data; recalibrate gates.

Order rationale: W1 cheap and unblocks everything; W2 before W3/W4 touch
checkers; W5 last validates modules AND rigor together. R3 is playbook text —
sprinkled across W1/W3. Proof-binding receipts stay backlog unless the
campaign surfaces drift incidents; revisit after W5.

## Non-goals
Pixel parity on Linux (P0 line, separate), prompt-injection guard, deadline
tracking (Z6 backlog). No OpenScience dependency — evaluation only. No
general-purpose planner — the resolver is a fixed-DAG topological walk.
