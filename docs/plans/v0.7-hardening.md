# Pipeline v0.7 — hardening plan (fail-closed, less human-dependent)

Lean core (gate integrity + Stage 4.5 + checkers) is implemented; the rest is
deferred-until-scale. Note: without release attestation this is **gate integrity,
NOT full fail-closed** — a direct-assembly bypass remains until the deferred release
verifier lands.

Source: a two-round design council between an orchestrator model and an independent
reviewer model, grounded in a real unattended run. This plan makes the AUTO/night
mode *enforce* what it currently only documents.

## Verified problem (why documentation wasn't enough)

An unattended run produced a finished document/PDF **outside the state machine**:

- The pipeline file showed every stage still `pending` and no canonical output, yet
  a deliverable was produced.
- The controller's gate command recorded a **caller-supplied integer** — it never
  ran the checker. A gate was "passed" with a typed 0.
- Night mode did not block on pending predecessor gates (only supervised mode did).
- The orchestrator (an LLM) took the path of least resistance: it called the COM
  backend directly, never the guarded fill entrypoint, and never advanced stages.
  **The controller was decorative.**

Conclusion: adding more checks without making transitions **fail-closed**, and
without making the compliant path the **easiest** path, will not reduce human
dependence.

## The four converged decisions

1. **Enforce at ACCEPTANCE, not production.** Production cannot be made
   unbypassable (COM, the COM library, the GUI, and raw filesystem access all
   exist). A workspace-internal manifest is *also forgeable by the orchestrator*.
   The only real choke point is a **release verifier that re-computes the whole
   chain from scratch and attests it**, running with authority the orchestrator
   lacks.
   - The assembly stage produces a *candidate* only, never a canonical output.
   - A separate release step (ideally out-of-workspace, or under a different OS
     principal) recomputes the sim results, claim, freeze, review, assembly, and
     final-review hashes; only it writes the release directory, the attestation,
     and the canonical-output pointer.
   - The operator submits only files whose hash matches a valid attestation.
   - **Environment ceiling (honest):** a repo with no real privilege boundary
     (orchestrator = full-filesystem shell) can reach **trust-by-recomputation** (a
     forged intermediate is caught because the verifier re-derives rather than
     trusts) plus the operator habit "submit from the release directory". A true
     cryptographic or OS-capability boundary is a later upgrade, not available
     today.
2. **One idempotent entrypoint `report_build <WS> --to release`.** This makes the
   guarded path the path of least resistance (otherwise the LLM routes around it,
   as it did). It locks the workspace, resumes from the earliest **unverified
   dependency** (not from the pipeline-file status), **runs each checker itself**
   (the public caller-supplied exit code is retired), captures each checker's
   path/version/command/exit/output-hash, drives write→humanize→resolve→assemble→QA
   →final review, calls the release step only when the whole dependency graph
   validates, and on a stop-line exits `20` with a human-required record. The
   pipeline file becomes a projection of the evidence graph, not the authority.
3. **Claim-binding token lifecycle** (kills number-drift without regex false
   positives):
   `sim → 2 numeric reviews → freeze results → tokenized draft → claim lint →
   protected humanize (tokens are opaque AST nodes; they cannot be deleted, moved,
   or retyped) → assert token invariants vs the pre-humanize AST → resolve ONCE →
   freeze resolved content → content review → assemble`. Tokens bind a rendered
   value to a results path, a derived expression, or a cited literature fact. A bad
   path is a **hard build error, never an empty render**. Numeric text does not
   exist during humanization at all — only the bindings do.
4. **Three human stop-lines** (night mode's "zero unplanned human rescues" target
   stays, but a *planned* stop is a success, not a rescue). Any single
   reviewer/preflight raising one **vetoes** — no consensus needed; every reviewer
   schema must return all three fields even when false:
   - **SENSITIVE FRAMING** — a normative or harmful claim about politics, protected
     groups, identifiable persons, medical/legal guidance, or a contested
     intervention. (Plain factual history does not trigger it.)
   - **LOAD-BEARING DISPUTE** — two sources or reviewers materially disagree on a
     thesis/method/result claim; one bounded adjudication round, else weaken,
     remove, or stop.
   - **UNSUPPORTED NOVELTY** — a load-bearing real-world claim with no literature
     support presented as fact (including a toy-model number transferred to the real
     system). Labeled hypotheses and results are fine.

## Delivered now (lean core, self-contained, tested)

- A deterministic content gate (recompute-based, exit-coded). HARD: no web
  citation, no prohibited endings, figure files exist, the summary deliverable is
  within its byte budget, the PDF has no equation-markup leak. WARN:
  parenthetical-English gloss, bare numeric reference labels, in-text vs
  bibliography mismatch. Tested to pass finished reports (zero hard findings) and to
  reject a planted negative (URL + prohibited ending + missing figure). This is the
  checker the build entrypoint and Stage 4.5 call themselves.
- A backend precheck → a run-capabilities file (CLI version minimums verified,
  live-ping optional). Council seating reads THIS, not a static status field. It
  reports a dual-pool or a degraded-multi-critic fallback mode.

## Sequence (remaining, staged so the large controller test suite isn't rushed)

1. Fix the controller transition semantics: a current-stage gate and required
   artifacts before `done`; night mode must honor script/receipt gates; a corrupt
   stages file is a stop (no silent embedded-graph fallback). **Retire the public
   caller-supplied exit code; the `check <ws> <gate>` subcommand runs the registered
   checker and records provenance.** (Plus update its test suite.)
2. Multi-gate stage contracts in the stages file; add **Stage 4.5 content_audit**
   (claims + style + figures + equations + curriculum + content review) before the
   assembly stage.
3. Manifests plus automatic invalidation: a sim freeze, a content freeze, and an
   assembly manifest; any upstream change invalidates Stage 4.5+ by hash.
4. The fill/assembly entrypoint refuses without a valid content-freeze manifest;
   products of a raw COM backend call stay uncertified candidates.
5. A council runner (a mandatory decision-council gate before the design stage is
   done; reads the run-capabilities file; a degraded quorum requires unanimity; a
   one-seat "council" is forbidden).
6. Claim tokens in the normalizer/build step + a claim linter + protected humanize.
7. A style checker (rules from the resolved packs), a source checker (Crossref/ISBN
   with a cache, not an LLM "looks real" judgment), and a curriculum checker.
8. An assembly render gate and a final release gate (fail on any blocking finding
   regardless of score); the return stage begins with release verification.
9. The single build entrypoint; an event ledger and a decision-policy file;
   regression tests reproducing every real human intervention and the re-assembly
   loop.

## Acceptance test for v0.7 (adversarial — each must FAIL without a human noticing)

Jump straight to assembly; pass a forged gate exit code; edit the results file after
freeze; kill an external reviewer before the council; change one caption after
review; deliver an unreviewed PDF. Each attempt must be rejected by
recomputation/attestation, not by a human catching it.

## Not hard gates (council rejected — unstable or gameable)

A generic AI-detector score; regex-only caption consistency; an LLM-only citation
"realness" check; live web search as the sole citation verifier; reviewer
file-count as council proof; a universal scientific-validity script; OCR-only
equation fidelity; automatic proof of author understanding.

# 10. Orchestration efficiency (converged two-round council, workflow-engine based)

A separate axis from enforcement: makes the agent workflow faster and cheaper
WITHOUT re-opening a bypass (cache hits are not acceptance; timeouts are not
verdicts; independence means a separate context; post-freeze edits invalidate
descendants; only the release step attests).

## Three-layer split (do NOT build a custom DAG runner — use the built-in workflow engine)

- **Workflow engine** = a pure-work DAG executor: parallel/pipeline, bounded
  concurrency, resume-from-cache, journal. Not an authority for gates, side effects,
  or release.
- **The build supervisor** = a thin host: input manifest, phase barriers, leases,
  subprocess supervision, evidence composition. Compiles the DAG into **epochs**
  separated by external barriers.
- **The release step** = the acceptance authority (recompute + attest).

Epochs: `A research/sim/design/review` → [human barrier if any] `B
write/content-audit/freeze` → [COM under an exclusive lease] `C visual/final review`
→ release.

## COM side-effect adapter (outside the workflow agents)

A named mutex; a unique scratch directory; a killable subprocess in an OS job
object; a hard deadline; on success, atomically promote to the candidate directory
and write an assembly receipt (input/content/ops/form hashes, tool and host
versions, produced document/PDF hashes, command/exit/timestamp). On timeout, kill
ONLY the owned process tree — **never blanket-kill the host word-processor
process** (an interactive user session may have it open). A dedicated OS user is a
stronger isolation upgrade later.

## Human wait = an epoch boundary (not a mid-node suspend)

Finish the epoch; write a human-required record bound to the input-manifest hash;
exit 20; on resume, validate an approval receipt against that hash; an upstream
change invalidates the approval. Within a repeated epoch, unchanged agent calls
still hit the workflow cache.

## Workflow cache identity

Every canonical agent call carries an identity tuple (task id, input-manifest hash,
policy hash, scope hash, schema version) in its prompt/options. Agents read
immutable hash-named snapshots, not a bare content file. Cached output is accepted
only if its receipt echoes the expected hashes and passes schema. The cache is
acceleration; the manifest plus release are authority.

## Agent-tool vs killable-CLI worker (liveness)

Reclaimable roles must be OS subprocesses that can be heartbeat-monitored,
timed-out, and killed; harness-controlled agent-tool subagents run opaquely (no
mid-run heartbeat, no kill) — the 15-minute stall was exactly that.

**A policy hook applies here:** the operator MAY ban a headless full-tool
orchestrator loop outright (a full-tool headless agent in a loop is arbitrary
command execution, recursive agent spawning, and quota/cost blowup risk). Where that
ban is in force, the killable-CLI workers are external review CLIs and scripts ONLY,
and the primary-model roles have no killable-CLI form → they always run via the
harness-controlled agent tool (which is not process-killable by the orchestrator).

```
killable CLI (external review CLIs / scripts — not the harness-controlled model) if:
   side-effecting  OR  (required_for_progress AND unattended)
   OR long/uncertain duration  OR  timeout/retry/backend-swap policy  OR  cost-bounded
agent tool (harness-controlled model roles; and optional/redundant work) if:
   this specific model is required  OR  the result is optional/redundantly covered
   (a permanent agent-tool stall must therefore not hold a gate or lease — so put any
    reclaim-critical model step behind a proven primitive plus a redundant/covered
    path, or make it a script/CLI step instead)
```

Consequence (honest): with a headless loop banned and agent-tool subagents
non-killable, **those model roles cannot be process-reclaimed.** Mitigate by:
proving the primitive before delegating; keeping reclaim-critical work on external
CLIs and scripts; never letting a single non-redundant agent-tool call be the only
thing holding a gate open. A supervisor heartbeat proves the supervisor is alive,
not that the LLM is making progress → also enforce an absolute deadline →
process-tree kill → backend swap. A missing reviewer quorum is `INFRA_BLOCKED`,
never a PASS. A stop-line survives retries (respawning until a reviewer says false
is verdict-shopping).

### Per-role budgets (no-progress / hard deadline)

precheck 20 s/45 s · mechanical 90 s/4 m · research shard 4 m/12 m · sim/figure
4 m/15 m · writer 4 m/12 m · numeric/content review 2 m/7 m · high-effort
council/final 4 m/12 m · proven COM 90 s/8 m.

## Model + effort tiering (roles that MUST stay a strong model at high effort: topic/framing critic, degraded council seats, final PDF adversarial, dispute adjudication)

No-LLM: DAG scheduling, hashes, cache, gates, manifests, cast-off arithmetic,
release verify. Cheap-model low effort: inventory/metadata/archive/wiki/task
summary, script error classification (cannot declare a gate). Mid-model high effort:
research shards, sim-from-spec, figures, contact-sheet visual QA. Strong-model high
effort: form/design judgment, main writer/content repair, evidence cross-examiner.
Two independent high-effort backends: the numeric reviewers A/B. Mid-model or
external CLI high effort: AI-tell/style (after the deterministic style checks).

## Cache boundary

A shared content-addressed store holds: content AST, claim bindings, figures, the
layout plan, ops, deterministic checker results. NOT shared (run/workspace only):
the assembled document, the exported PDF, contact sheets, render receipts — COM
export is not byte-deterministic, so input-keyed reuse would serve whichever
arbitrary render was stored first (dangerous). Same-build checkpoint reuse is
allowed only if the assembly receipt and the actual hashes validate; a clean build
or a release recomputation reruns COM. Downstream reviews bind to the ACTUAL
produced PDF hash, never a predicted one.

## Incremental review (cut the dominant review-token cost, kept sound)

1. Deterministic full-document checkers first (free): sim recompute + invariants,
   claim-token binding, citation/source registry, style/prohibited-pattern,
   figure/table/equation reference integrity, content↔ops coverage, PDF leak plus
   layout metrics.
2. LLM reviewers get only the residual semantics: method appropriateness,
   sign/convention, causal overreach, model-limit interpretation, framing/curriculum,
   source disagreement, unsupported novelty.
3. Build a review graph (source/result → claim id → paragraph → section conclusion →
   abstract/thesis → dependent caption/table); each unit has a stable id and hash;
   each receipt records the unit hashes, the dependency-closure hash, the lens, the
   policy, and the verdict.

Re-review after a fix: AST diff → transitive closure. Stay **LOCAL** (re-review only
the changed spans and immediate dependents) iff the change stays within existing
unit ids, adds or removes no claim/citation/result/figure/dependency edge, does not
touch the title/abstract/question/method/results/limitations/conclusion/framing,
closes within one section, still passes the checkers, is classified LOCAL (not
UNKNOWN), and its post-assembly pagination is confined. Otherwise do a **full**
re-review. The final candidate always gets at least one fresh full adversarial pass
on the stabilized PDF; the release binds the composite receipt set to the exact
final PDF hash. This preserves the two-reviewer numeric contract while deterministic
recomputation handles arithmetic and binding.

## Top three orchestration wins per effort

1. Stage 4.5 review/freeze before assembly + automatic hash invalidation (the
   biggest COM saving, and also enforcement).
2. A workflow-based DAG (built-in engine) with backend preflight, leases,
   heartbeats, and fenced attempts (kills silent stalls, stale-model failures, and
   coordination-context cost).
3. A content-addressed node cache for sim/figures/ops/checkers (NOT COM/PDF).

Review batching is fourth (dispatch and lenses batched; reviewer identities never
merged into one context).

# 11. Reality check — practitioner data + new-model economics

The §1–10 design was reasoned from first principles plus one run's evidence, NOT
grounded in current practitioner experience or new-model economics. Contemporary
web research both validates the DIRECTION and exposes over-engineering. Honest
correction:

**Validated (keep):** a hybrid — a deterministic workflow orchestrating agentic
nodes for specific tasks — is the practitioner-consensus "best results" pattern; a
kill switch plus structured logging are mandatory; human-in-the-loop
pause-and-resume with serialized state and a checkpoint (interrupt-style) is our
epoch-boundary human-wait; a graph-defined flow (not free agent choice). Upcoming AI
regulation makes demonstrable human oversight a LEGAL requirement → the three
stop-lines are compliance, not just safety.

**Challenged (change emphasis):**

- **Centralized multi-agent orchestration carries a large token overhead; given
  equal compute, a single strong agent often matches or beats a multi-agent system
  on REASONING tasks.** With a top model's very large context and strong reasoning,
  many things a run fans across agents are better as one strong pass. → Do NOT
  multi-agent for reasoning quality. Reserve multi-agent strictly for (a) genuine
  INDEPENDENCE (the two numeric reviewers — a different rationale than reasoning)
  and (b) DOMAIN decomposition (research / sim / figures are distinct tool domains).
  Everything else: fewer, stronger, more deterministic.
- **Cost:** agentic orchestration is markedly more expensive per month than a
  deterministic workflow at scale; the orchestrator's decompose/aggregate LLM calls
  are the hidden cost. → maximize deterministic code (the content verifier, the
  checkers), minimize LLM round-trips (this strongly reinforces incremental review,
  §10).
- **Specification and planning failures dominate; cascading errors are the top
  bottleneck** → the sim spec and design spec quality is the highest-leverage
  investment; a fail-closed freeze plus invalidation directly targets cascades.

**New-model tiering (empirical):** the strongest reasoning model with the largest
context is best on cascade-sensitive multi-file work but the most expensive → reserve
it for the writer, the final adversarial pass, and the framing critic. A top
hard-engineering model is best for building the hardening scripts themselves. A
large-tool-surface model is best for checker-driving and code review. A fast,
cheaper, strong-fresh-code model is the cost-efficient second numeric reviewer.

**Scale caveat (the honest one):** for the ACTUAL workload (one author, a few
reports at a time), the full custom DAG-runner plus release-verifier machinery is
likely OVER-BUILT. Default to the **lean form**: (1) deterministic code maximized;
(2) one strong pass where a run would otherwise fan out; (3) multi-agent only for
independence plus domain decomposition; (4) a kill switch, human-in-the-loop, and
logging. Escalate to the full machinery only if this ever scales to many reports,
multiple machines, or multi-day waits.
