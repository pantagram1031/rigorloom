# v0.10 — Typeset parity without Hancom + Studio/Linux integration

Status: IN PROGRESS (2026-07-14). Operator pre-approved. Follows v0.9.

Measured gap (v0.9 close): the XML engine matches COM output on content, numbers,
equations, figures and page count, but the COM path's render-measured FILL loop
(page fill, gap removal, ±line deltas) gives it a typesetting edge — the XML path
assembled in one pass because it had no renderer to measure against. v0.10 gives
the XML path the same loop using any available renderer, calibrated against real
Hancom output.

## Y1 — Renderer-measured FILL loop for the XML engine (hwp-master)

- fill_report `--engine xml` gains loop/measure modes: when a renderer is
  configured (`--pdf-cmd` argv template, e.g. LibreOffice+H2Orestart headless),
  each iteration renders the assembled hwpx and runs layout_qa on the PDF,
  emitting the SAME verdict contract as the COM loop (gappy pages, bottom_white,
  needs[]) so the orchestrating agent can apply content deltas and re-enter.
- No renderer → current single-pass `xml_verified_no_proof` behavior unchanged.
- Verdicts carry `proof_grade: hancom|advisory|none` so downstream consumers and
  the pipeline know which renderer measured them.

## Y2 — Cross-renderer calibration (evidence, not vibes)

- `scripts/render_calibrate.py`: render the SAME hwpx via Hancom COM and via a
  second renderer (LibreOffice+H2Orestart — on this machine through WSL), run
  layout_qa on both PDFs, and report metric deltas (page count, per-page
  bottom_white, max_gap, text-length drift). Output: calibration JSON + a
  human-readable table.
- Run it on a real finished document; commit the generalized findings (tolerance
  offsets, known divergences — e.g. font metric differences) to
  `references/render_calibration.md` (numbers only, no report content).
- The advisory FILL loop applies these tolerances so LO-measured verdicts don't
  chase Hancom-invisible ghosts.

## Y3 — Renderer discovery + capability probe (rigorloom)

- `pipeline/scripts/render_probe.py`: detects available document capabilities on
  this machine — Hancom COM (pyhwpx importable + Hwp registry), soffice on PATH,
  soffice via WSL (`wsl -e soffice`), H2Orestart presence — and emits
  `capabilities.json` (argv templates for each usable renderer).
- doc_backend hwpx tier consumes it: auto-passes the best `--pdf-cmd` so Linux
  machines get the advisory proof loop with zero manual config.
- bootstrap.py prints the capability matrix at the end of its summary.

## Y4 — Studio integration (the website speaks Linux)

- Per-workspace capability strip: which backends/renderers this machine offers
  (from render_probe, cached), and which proof grade the current output carries.
- ACTION MODE gains `build-hwpx` (doc_backend --backend hwpx) alongside
  build-bundle; deliverable panel shows the advisory PDF when present, labeled
  with its proof grade (never presented as submit-grade when advisory).
- Verdict panel renders the xml-loop verdict (proof_grade, gappy pages, needs)
  so the operator sees typeset quality state without opening files.

## Measured limitation (2026-07-14, WSL bisect)

H2Orestart 0.7.13 + LibreOffice 24.2 CRASHES ("Unspecified Application Error")
on any hwpx containing `<hp:equation>` — confirmed by zip-surgery bisection on
two independent real documents (removing equations alone makes both convert;
pics/tables/BinData are innocent). Consequence: the LibreOffice advisory-proof
loop only works on equation-FREE documents, which excludes most real reports.

Handling:
- render_probe / doc_backend detect equations in the target (grep section XML
  for `<hp:equation`); if present AND the only renderer is LibreOffice, the
  advisory loop is marked `proof_grade: none` with reason `renderer_cannot_eqn`
  — never a false advisory pass on a doc the renderer mangles.
- The FILL loop still runs structurally (layout_qa on the parts LO renders) only
  when the doc is equation-free; otherwise single-pass xml_verified_no_proof.
- Calibration numbers (references/render_calibration.md) are from the
  equation-stripped subset, n=2, explicitly caveated. Advisory tier = layout
  sanity for equation-free docs; equation docs need Hancom for any visual proof.
- Upstream watch: H2Orestart issue tracker (equation import) — revisit when a
  fixed release lands.

Non-goals: making LibreOffice output submit-grade (final submission still gets a
Hancom pass where available — calibration narrows, does not erase, renderer
differences); .hwp binary editing; rendering equation-bearing hwpx without Hancom
(blocked upstream in H2Orestart).
