# Example layout conventions

These are optional examples, subordinate to the request and form.

- Keep body-page density reasonably uniform.
- Prefer inline equations when they remain readable; use display equations for
  deliberate emphasis or multi-line derivations.
- Keep table rows compact and allocate more width to explanatory columns.
- Captions should be visually subordinate to body text.
- A heading at the bottom is not automatically a defect; a heading followed by
  a large unexplained void is.
- Validate layout through a contact sheet before inspecting individual pages.
- **Figures collected at the end.** Rather than interleaving figures with the
  body, one example convention groups every figure into a final "Figures"
  section; the body references them by number ("as shown in Figure n"). Each
  figure's caption reads "[Figure n] <description>" placed below the figure,
  in body-text size. Do not also embed a title or number inside the figure
  image itself — a caption in the final section AND a label baked into the
  image is double captioning and should be avoided.
- **Page-number footer.** A simple centered "- n -" footer, numbering from
  the first body page rather than the cover or a preliminary guidance page.
- **Tight paragraphs, bold inline sub-headings.** Minimize blank lines
  between paragraphs. When a subsection needs a label, a bold inline
  sub-heading such as "(1) Title" that leads directly into the paragraph
  reads as more natural than a separate heading block with its own spacing.
- **Guide-table keep-vs-delete is an operator decision.** Form templates
  sometimes ship a guidance table or notice with an instruction to delete it
  before submission. Whether to follow that instruction or keep the table as
  submitted is a judgment call, not a default the pipeline should silently
  make — surface it as an explicit choice and record which way it went.
